# Architecture Decisions

Every decision made during design, with rationale. Do not re-litigate these.

## Web App, Not Native Desktop (Tauri)

**Decision:** Browser-based wizard generating a .zip file.
**Rationale:** Unsigned desktop apps show 7 warning steps on Windows (SmartScreen) and 6-step bypass on macOS (Gatekeeper). Non-technical users abandon at the first warning. Code signing costs $99-220/year. A web app has zero friction — visit URL, fill wizard, download zip.
**What we lose:** Auto-install of prerequisites (Node.js, Git, Claude Code). Solved by Getting Started PDF included in the zip.
**What we gain:** Zero-cost distribution, works on every OS, instant updates, lower dev cost.

## Client-Side Only

**Decision:** No backend server. All generation happens in the browser.
**Rationale:** `client-zip` generates zips in-browser (2.6KB). `@react-pdf/renderer` generates PDFs in-browser. Telemetry is a single POST. Hosting is free static site on Vercel.
**Implication:** No server costs, no infrastructure, no auth, no data storage beyond telemetry webhook.

## React + Vite + TypeScript + Tailwind

**Decision:** React for the wizard UI.
**Rationale:** Best multi-step form libraries (React Hook Form + Zod for validation). Single FormProvider wraps all steps, preventing cross-step data loss. Tailwind for styling. Vite for fast dev/build.

## @react-pdf/renderer for PDF

**Decision:** React components → PDF blob → included in zip.
**Rationale:** Runs entirely client-side, no headless browser needed. Supports images (Praton logo), code blocks (styled monospace), clickable links. Professional output. 200-500KB for a 5-10 page guide.

## No Stoneforge

**Decision:** Stoneforge is not included in generated projects.
**Rationale:** Tested extensively in MYCOS. Failed on VPS due to env var propagation. Even local install adds overhead that isn't justified for sequential workflows. The real value (fresh context per task, auto-fix on failure) doesn't outweigh the complexity. Sequential rotation queues work fine. May revisit in future versions.

## No Custom Subagents

**Decision:** Not generated by default.
**Rationale:** Claude Code itself pushed back during MYCOS sessions. For solo developers with serial workflows, maintaining subagent definitions costs more than the $0.02/invocation savings. Create subagents only when parallel agents need specialized roles.

## No Agent Hooks (Command Hooks Only)

**Decision:** All hooks use `type: "command"`, not `type: "agent"`.
**Rationale:** A deterministic command hook for binary pass/fail checks (tests pass? docs updated?) is more reliable and cheaper than an LLM evaluating the same thing. Agent hooks add token cost for no benefit on binary checks.

## PostToolUse on Git Commit, Not Stop Hook

**Decision:** Test and doc enforcement triggers on `git commit`, not on the Stop event.
**Rationale:** The Stop event fires every time Claude Code ends a turn — running SSH + pytest after every response made Claude Code unusable in MYCOS. PostToolUse with Bash matcher detecting `git commit` catches the exact moment that matters (code being persisted) without blocking every interaction.

## Session-Start for Growth Checks

**Decision:** CLAUDE.md line count, module count, and frontend file change detection run in session-start.sh, not stop-gate.
**Rationale:** Growth checks are informational, not blocking. They belong at session start where Claude Code sees the warnings once and can act during the session. Stop hooks should only enforce quality gates.

## File Generation Philosophy

**Decision:** Every generated file must have a consumer on day one.
**Rationale:** From MYCOS experience: empty template files get ignored for weeks, waste context when Claude Code reads them, and the template structure is wrong by the time you need them. Generate files with real content or don't generate them at all. The wizard questionnaire provides real content (project description, goals, design principles) that seeds files with useful data from day one.

## Two-Tier CLAUDE.md Only When Complex

**Decision:** Simple projects get one CLAUDE.md. Complex projects (DB + VPS + frontend + backend) get CLAUDE.md + .claude/CLAUDE.md.
**Rationale:** A fresh project with a 60-line CLAUDE.md doesn't need a second tier. The session-start hook detects when CLAUDE.md exceeds 150 lines and warns Claude Code. The Self-Maintenance section in CLAUDE.md tells Claude Code how to split. For complex projects, the questionnaire pre-generates the two-tier structure.

## Telemetry Design

**Decision:** Anonymous, consent-gated (proceed/exit, no skip), questionnaire answers only.
**Rationale:** No email capture — download users aren't paying customers. The questionnaire answers themselves are the valuable dataset (what people build, what infrastructure they use). No IP logging, no fingerprinting. GDPR-compliant since anonymous + consensual. Webhook to Supabase table.

## Launcher Scripts

**Decision:** Generate OS-specific double-clickable scripts for starting Claude Code.
**Rationale:** Non-engineers don't know how to open a terminal. Launcher scripts eliminate that barrier. Multiple variants: standard, full permissions, plan mode, cost report. The filename IS the documentation.
